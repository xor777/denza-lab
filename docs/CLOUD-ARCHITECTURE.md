# Car ADB Gateway — архитектура cloud-режима (v2, после ревью)

Продукт generic, не привязан к марке: **Car ADB Gateway** — Android-приложение для автомобильных головных устройств (ГУ), где ADB доступен локально приложениям, но не наружу. Denza Z9GT — первое поддерживаемое/протестированное устройство. Текущий код (`dev.denza.adbbridge`, denza-gateway) подлежит ребрендингу — §7.

Документ описывает cloud-режим: доступ к ADB ГУ через публичный интернет, через мультитенантный relay-сервер, с управлением доступами. LAN-режим сохраняется.

История: v2 учитывает инженерное ревью — раздельные relay-аккаунты, PAM-enrollment, host-key pinning, отвязка от Wi-Fi, FGS-лайфсайкл, pairing-once вместо статического кода.

## 1. Принципы

1. **ГУ всегда инициирует соединение наружу** (reverse tunnel). Никаких входящих портов на машине.
2. **Relay — слепой коммутатор.** Пробрасывается порт встроенного SSH-сервера приложения (2222), а не ADB. Клиент проходит E2E SSH до самой машины.
3. **Два слоя аутентификации:** ключи на relay (кто может дотянуться до порта) + доверие самой машины (pairing-once, §2.4.4).
4. **Минимум серверного кода.** Relay = стоковый OpenSSH + PAM-скрипт + два shell-скрипта. Мультитенантность = Unix-аккаунты + опции ключей в `authorized_keys`.
5. **Ключи не переносятся руками.** Приватные ключи рождаются там, где используются, и не покидают устройство. Публичные ключи регистрируются автоматически (enrollment по одноразовому коду) или одной командой админа.
6. **UX-контракт (§3) — часть спецификации.** Реализация, требующая от пользователя больше действий, чем описано в §3, считается несоответствующей.

## 2. Компоненты

```
┌─────────────┐   reverse SSH (-R)    ┌──────────────┐   SSH (-W)      ┌──────────────┐
│  ГУ (авто)  │ ─────────────────────▶│ Relay (VPS)  │◀─────────────── │ Мак клиента  │
│ gateway app │  исходящее, порт 443  │ OpenSSH +    │  исходящее      │ cag CLI      │
│ sshd :2222  │  (сотовая сеть)       │ PAM + скрипты│                 │ ssh + adb    │
└─────────────┘                       └──────────────┘                 └──────────────┘
      внутри туннеля: клиент ⇄ (E2E SSH, pairing-once) ⇄ sshd приложения ⇄ ADB
```

### 2.1 Relay-сервер (VPS)

Стоковый OpenSSH, слушает **порт 443** (исходящий 22 часто блокируется в гостевых/корпоративных сетях; 443 почти всегда открыт). Админ ходит своим обычным пользователем (можно оставить и порт 22 для него).

**Три сервисных Unix-аккаунта** — направления форвардинга разделены на уровне аккаунтов, потому что пер-ключевые опции этого не дают (`permitopen` ограничивает только `-L`, `permitlisten` — только `-R`, а `port-forwarding` включает оба направления; `restrict` не запрещает исполнение команд без PTY):

```
# /etc/ssh/sshd_config.d/cag.conf
Port 443
PasswordAuthentication no
PermitRootLogin no

Match User cag-device
    AllowTcpForwarding remote
    PermitTTY no
    X11Forwarding no
    AllowAgentForwarding no
    ForceCommand /bin/false

Match User cag-client
    AllowTcpForwarding local
    PermitTTY no
    X11Forwarding no
    AllowAgentForwarding no
    ForceCommand /bin/false

Match User cag-enroll
    PasswordAuthentication yes
    AllowTcpForwarding no
    PermitTTY no
    X11Forwarding no
    AllowAgentForwarding no
    ForceCommand /opt/cag/enroll.sh
```

- **`cag-device`** — устройства. Только `-R`, каждое — на свой порт: `~cag-device/.ssh/authorized_keys`:
  ```
  permitlisten="127.0.0.1:10001" ssh-ed25519 AAAA... device:car-dmitry
  permitlisten="127.0.0.1:10002" ssh-ed25519 AAAA... device:car-vasya
  ```
- **`cag-client`** — люди. Только `-L`/`-W`, каждый — на разрешённые порты: `~cag-client/.ssh/authorized_keys`:
  ```
  permitopen="127.0.0.1:10001" ssh-ed25519 AAAA... user:dmitry
  permitopen="127.0.0.1:10002" ssh-ed25519 AAAA... user:vasya
  ```
  Несколько устройств = несколько `permitopen` через запятую у одного ключа.
- **`cag-enroll`** — только регистрация устройств (§2.2). Forwarding запрещён полностью.

Итог: клиентский ключ физически не может сделать `-R` (сесть на порт офлайновой машины и выдать себя за неё), ключ устройства не может сделать `-W` (постучаться к чужой машине), никто не может исполнить команду.

Порты устройств слушаются только на `127.0.0.1` relay, наружу не видны.

**Состояние сервера** (файлы, не БД; источник истины — state, оба `authorized_keys` — производные, перегенерируются атомарно: временный файл + `mv`, под `flock`):
- `/opt/cag/state/devices.tsv` — имя → порт → fingerprint туннельного ключа → fingerprint внутреннего host key → дата;
- `/opt/cag/state/users.tsv` — имя → pubkey → дата;
- `/opt/cag/state/grants.tsv` — user → device → дата;
- `/opt/cag/state/codes/<CODE>` — активные коды привязки (имя устройства, зарезервированный порт, expiry).

**fail2ban**: джейл на неудачные аутентификации `cag-enroll` (стандартный sshd-фильтр — благодаря PAM неверный код это обычный auth failure в логе sshd).

### 2.2 Enrollment: PAM + ForceCommand

Проверка кода происходит **на стадии аутентификации через PAM** — sshd не пропустит произвольный пароль системного пользователя, поэтому вариант «код как пароль + проверка в скрипте» не работает; PAM — обязательная часть дизайна, не альтернатива.

PAM-стек sshd (фрагмент; для всех, кроме `cag-enroll`, ветка обходится):

```
auth  [success=1 default=ignore]  pam_succeed_if.so  user != cag-enroll
auth  [success=done default=die]  pam_exec.so expose_authtok quiet /opt/cag/check-code.sh
```

- `check-code.sh` (PAM): читает пароль (код) со stdin (`expose_authtok`), проверяет существование файла `state/codes/<CODE>` и TTL. Успех → аутентификация пройдена. Код здесь **не** сжигается (PAM может вызываться повторно до успеха сессии).
- `enroll.sh` (ForceCommand, запускается после успешной аутентификации): протокол через stdin/stdout, всё построчно:
  - stdin строка 1: код (повтор — чтобы скрипт знал, какой код потреблять);
  - stdin строка 2: публичный **туннельный** ключ устройства (OpenSSH-формат);
  - stdin строка 3: публичный **host key внутреннего sshd** устройства (для pinning на клиентах, §2.5);
  - действия под `flock`: валидация обоих ключей (`ssh-keygen -l`), проверка и удаление файла кода (одноразовость), запись в `devices.tsv`, добавление строки в `authorized_keys` пользователя `cag-device` с `permitlisten` на зарезервированный порт;
  - stdout: `OK <port>` либо `ERR <reason>` (`invalid-code`, `bad-key`, `conflict`).
- Аккаунт `cag-enroll`: системный пароль залочен корректно для PAM-обхода (`usermod -p '*'`), shell `/usr/sbin/nologin` не ставить (ForceCommand нужен запуск скрипта) — использовать обычный shell + ForceCommand.

### 2.3 CLI `cag-admin` (сервер)

Один bash/python-скрипт, работает со state-файлами, перегенерирует `authorized_keys` атомарно.

```
cag-admin device invite <name>     # резервирует порт, создаёт код (XXXX-XXXX, алфавит без
                                   # неоднозначных символов, TTL 60 мин), печатает готовое
                                   # сообщение для отправки другу: адрес relay + код
cag-admin device revoke <name>
cag-admin user add <name> --key-file <pub>|--key "<pubkey>"
cag-admin user revoke <name>
cag-admin access grant <user> <device>   # добавляет permitopen; печатает grant-bundle (см. ниже)
cag-admin access revoke <user> <device>  # убирает permitopen; ПЕЧАТАЕТ ПРЕДУПРЕЖДЕНИЕ:
                                         # уже установленные сессии не рвутся (kill-sessions — не MVP)
cag-admin status                   # устройства (порт, online/offline по ss/netstat), пользователи,
                                   # гранты, активные коды
```

Временные гранты (`--for 24h`) в MVP **не входят**: удаление ключа не завершает живые SSH-сессии и создаёт ложное ощущение немедленного отзыва.

**grant-bundle** — одна base64-строка (JSON): `{alias, relayHost, relayPort, relayHostKey, devicePort, innerHostKey}`. Отправляется пользователю в мессенджер, потребляется `cag import` (§2.5). Содержит всё для строгого pinning — пользователь не настраивает ничего руками.

### 2.4 Android-приложение: cloud-режим

#### 2.4.1 Ключи устройства
Два независимых ключа, оба генерируются на устройстве и не покидают его:
- **туннельный ключ** (ed25519, новый) — аутентификация на relay;
- **host key внутреннего sshd** (уже существует, `hostKeyFile`) — E2E-идентичность машины.
Enrollment регистрирует публичные части обоих.

#### 2.4.2 Два независимых листенера
Существующий `SshGatewayServer` рассчитан на один bind-адрес — нужен общий core и два экземпляра/листенера с разными политиками:
- **LAN-листенер**: Wi-Fi IP + существующая subnet-политика. Живёт только при активном Wi-Fi (как сейчас).
- **Cloud-листенер**: только `127.0.0.1:2222`, принимает соединения только с loopback. **Не зависит от Wi-Fi** — работает на сотовой сети; `NetworkInfoProvider` больше не является гейтом запуска сервиса, а только LAN-части. `0.0.0.0` не слушать.

#### 2.4.3 Enrollment и туннель
- **Enrollment-клиент** (MINA SSHD client, уже в зависимостях): подключение `cag-enroll@relay:443`, password = код, exec-канал, протокол §2.2. Успех → сохранить порт, `enrolled=true`. Host key relay: TOFU на этом первом соединении, далее строгий pinning (принятое остаточное допущение — §5).
- **`CloudTunnelClient`**: исходящее `cag-device@relay:443` по туннельному ключу, строгая проверка запиненного host key relay; remote forwarding `127.0.0.1:<port>` → `127.0.0.1:2222`; keepalive 30 с; reconnect с экспоненциальным backoff (1 с → 60 с cap) на любой обрыв и смену сети.

#### 2.4.4 Pairing-once (замена статического pairing-кода)
Проблема текущей модели: код генерируется при старте процесса и живёт до ручной ротации — как долгоживущий пароль это слабо, а с автортацией владелец не смог бы подключаться удалённо (код виден только на экране машины). Решение — модель самого ADB: **код используется один раз для доверия компьютеру, дальше — ключи**.

- Внутренний sshd получает **pubkey-аутентификацию** по app-managed списку доверенных клиентских ключей (`trusted_clients`, файл в app storage) вместо `RejectAllPublickeyAuthenticator`.
- **Механика pairing**: клиент подключается обычным ssh со своим ключом; sshd записывает предложенный (но ещё не доверенный) pubkey в контексте сессии, отклоняет pubkey-auth, клиент падает в password-auth; если введён валидный pairing-код — записанный pubkey добавляется в `trusted_clients`, сессия продолжается. Со второго подключения pubkey-auth проходит сразу, без ввода чего-либо.
- **Pairing-код**: существует только пока владелец держит «окно» открытым — кнопка **«Открыть доступ»** в UI показывает крупный код; TTL 10 мин; успешный pairing закрывает окно; cooldown/lockout после N неверных попыток (защита от брутфорса из туннеля). Password-auth вне открытого окна отключён полностью.
- **UI**: список «Доверенные компьютеры» (комментарий ключа + дата) с удалением; кнопка «Открыть доступ».
- Применяется к обоим листенерам (LAN и cloud) — единая модель.

#### 2.4.5 Лайфсайкл 24/7
- Текущий тип FGS `dataSync` на Android 15+ ограничен ~6 ч/сутки. Реальные ГУ часто на более старых Android — **проверить на устройстве**. Целевой тип: `connectedDevice`; если не проходит по политикам платформы устройства — `specialUse` (приложение распространяется APK, не через Play).
- Автозапуск после reboot: `RECEIVE_BOOT_COMPLETED` → поднять сервис, если cloud-режим был включён (настройка-тумблер).

#### 2.4.6 Экран Cloud (в терминах §3)
Основной экран: тумблер «Удалённый доступ», статус («подключено / нет интернета / переподключение»), имя машины, кнопка «Открыть доступ» (показывает pairing-код). Первичная настройка: два поля — адрес relay, код привязки → кнопка Enroll. Всё остальное (порт, fingerprint'ы, endpoint mode, логи) — под «Дополнительно».

### 2.5 Клиент (мак): CLI `cag`

Один bash-файл, распространяется через репозиторий (`curl -o` в README / Homebrew formula позже).

```
cag init                    # генерирует ~/.config/cag/id_ed25519 (если нет), печатает pubkey
                            # с подписью «отправьте администратору relay»
cag import <grant-bundle>   # разбирает bundle: пишет конфиг машины, pin relay host key и
                            # inner host key в отдельный ~/.config/cag/known_hosts
cag connect <alias>         # см. ниже
cag list                    # известные машины
```

`cag connect <alias>` делает всё сам:
1. выбирает свободный локальный порт;
2. строит цепочку `ssh -W` через relay (ключ из `cag init`) → внутренний ssh к машине (`StrictHostKeyChecking=yes`, свой `known_hosts` из bundle, `HostKeyAlias=<alias>`);
3. первый раз к данной машине: просит ввести pairing-код («попросите владельца нажать „Открыть доступ“»); далее — pubkey, без ввода;
4. определяет endpoint (smart socket 5037 / raw adbd 5555) и поднимает соответствующий `-L`;
5. открывает subshell с готовым окружением (`ADB_SERVER_SOCKET`/алиас) либо `cag connect <alias> -- adb shell`;
6. по выходу гасит туннель.

Пользователь не видит `ProxyCommand`, порты relay, `-L`, `adb -H` и разницу 5037/5555.

## 3. UX-контракт

Это нормативная часть. Каждый сценарий — полный список действий пользователя, ничего сверх.

**Друг подключает свою машину (один раз):**
1. Получает в мессенджере от админа: адрес relay + код (`relay.example.com` / `7K3M-92QF`).
2. В приложении на ГУ: вводит адрес и код → «Enroll» → включает «Удалённый доступ».
3. Больше никогда ничего не делает (автозапуск, reconnect — сами).

**Разработчик, первичная настройка мака (один раз):**
1. `cag init` → отправляет напечатанный pubkey админу.
2. Получает от админа grant-bundle → `cag import <строка>`.

**Разработчик подключается к своей машине (каждый раз):**
1. `cag connect car-mine` → adb работает.
   Самый первый раз — плюс один шаг: нажать «Открыть доступ» на экране машины и ввести код (вы в этот момент в машине или рядом). После этого — никогда.

**Коллега просит помочь с его машиной:**
1. Админ (вы): `cag-admin access grant dmitry car-vasya` (если гранта ещё нет).
2. Коллега в машине жмёт «Открыть доступ», диктует код.
3. Вы: `cag connect car-vasya`, вводите код — его машина запомнила ваш мак. Коллега в любой момент удаляет ваш ключ из «Доверенных компьютеров».

**Админ:** одна команда на событие (invite / add / grant / revoke / status).

## 4. Флоу (протокольный уровень)

### 4.1 Регистрация устройства
```
админ:   cag-admin device invite car-vasya → порт 10002 зарезервирован, код 7K3M-92QF (TTL 60 мин)
друг:    вводит адрес + код на ГУ
ГУ:      ssh cag-enroll@relay:443 (password=код, PAM-проверка)
         stdin: код, tunnel-pubkey, inner-host-pubkey → stdout: OK 10002
ГУ:      сохраняет порт, поднимает туннель; код сожжён
```

### 4.2 Рабочее подключение
```
ГУ:      ssh cag-device@relay:443, -R 127.0.0.1:10002 → 127.0.0.1:2222   (висит, reconnect+backoff)
клиент:  cag connect car-vasya
         = ssh -W 127.0.0.1:10002 cag-client@relay:443  (ProxyCommand, pinned relay key)
           → ssh gateway@car-vasya:2222                 (pinned inner host key, pubkey auth)
           → -L <localport>:127.0.0.1:5037
         → adb через localport
```

### 4.3 Pairing нового компьютера
```
владелец: «Открыть доступ» → код на экране (TTL 10 мин)
клиент:   cag connect → pubkey отклонён → password=код → pubkey добавлен в trusted_clients
дальше:   всегда pubkey, без ввода
```

## 5. Модель угроз

| Угроза | Защита |
|---|---|
| Relay скомпрометирован **после** enrollment | Видит только шифротекст; клиент пинит inner host key из grant-bundle → MITM невозможен; ADB недостижим |
| Relay скомпрометирован **с самого начала** | Принятое остаточное допущение self-hosted модели (bundle формируется из state relay). Параноидальный путь: `cag connect --verify-fingerprint` сверяет inner fingerprint с экраном машины (owner диктует) |
| Сетевой MITM при enrollment | Окно = TTL кода (минуты), нужен in-path атакующий на сотовом канале; далее — строгий pinning. Принятый остаточный риск |
| Утечка кода привязки | Одноразовый, TTL, даёт только permitlisten на заранее выделенный порт |
| Брутфорс enroll | PAM → неудачи видны sshd-логу → fail2ban; ~40-битные коды, TTL |
| Клиент лезет к чужой машине | `permitopen` + `AllowTcpForwarding local` (не может «-R»-нуть чужой порт и имперсонировать устройство) |
| Устройство лезет к чужим портам/relay как прокси | `AllowTcpForwarding remote` + `permitlisten` (не может «-W») |
| Исполнение команд на relay | `ForceCommand /bin/false` на всех сервисных аккаунтах |
| Брутфорс pairing-кода из туннеля | Password-auth только при открытом окне (TTL 10 мин) + cooldown/lockout |
| Утечка клиентского ключа | `user revoke` + удаление из «Доверенных компьютеров» на машинах |
| Утечка туннельного ключа устройства | Даёт только публикацию порта; `device revoke` |
| Отзыв ≠ разрыв живой сессии | Явно задокументировано в `access revoke`; kill-sessions — за рамками MVP |

## 6. Этапы реализации

1. **Ребрендинг** (§7) — до нового кода, чтобы cloud-код писался в новых неймспейсах.
2. **Relay + узкий E2E-прототип**: `setup.sh` (аккаунты, sshd-конфиг, PAM, fail2ban), `check-code.sh`, `enroll.sh`, `cag-admin`. Прототип без приложения: обычный `ssh -R` с ноутбука изображает устройство → проверить всю цепочку relay → loopback sshd → реальный `adb`. Тесты скриптов в Docker (bats/sh).
3. **Приложение, core**: туннельный ключ, enrollment-клиент, рефакторинг на два листенера, `CloudTunnelClient` (reconnect/backoff), FGS-тип + boot autostart, минимальный UI (адрес+код, тумблер, статус). Unit-тесты: протокол enroll, backoff, политика loopback-листенера.
4. **Pairing-once**: pubkey-auth + trusted_clients, окно «Открыть доступ», cooldown, UI списка доверенных. Применить и к LAN-режиму.
5. **Клиент**: `cag` CLI (init/import/connect/list), генерация grant-bundle в `cag-admin`.
6. **E2E на железе**: реальное ГУ, сотовая сеть, VPS; проверка лимитов FGS на прошивке.

Серверная часть — директория `relay/` в этом же репозитории.
